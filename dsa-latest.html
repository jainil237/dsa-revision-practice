Pattern 01

Sliding Window

Where Used

Contiguous subarrays/substrings — finding max/min length, max sum, or a specific condition within a window of elements.

Hints to Identify

"contiguous""subarray""substring""longest/shortest""maximum/minimum""at most K"
What is Sliding Window?

Imagine you have a magnifying glass and you're scanning a line of text. The magnifying glass shows you only a contiguous portion of the text at any time. You can slide it right to see the next portion, or shrink it from the left to focus on fewer characters.

That's exactly what sliding window does programmatically. You maintain a window defined by two indices: left and right. The window represents a contiguous subarray/substring. Instead of recalculating everything for every possible subarray (which is O(n²)), you reuse the work from the previous window by only adjusting what changed at the edges.

Array: [a, b, c, d, e, f, g]
           [===window===]    ← left=1, right=3, window=[b,c,d]
                 [===window===]  ← slide right: left=1, right=4, window=[b,c,d,e]
              [===window===]    ← shrink left: left=2, right=4, window=[c,d,e]

The key insight: when you slide right by one position, you're only adding one new element and possibly removing one old element. You don't need to reprocess the entire window — just update the state incrementally. This turns O(n²) into O(n).

When to Use • When NOT to Use
Use Sliding Window When
  • Problem asks about a contiguous subarray/substring
  • You need the longest/shortest window satisfying a condition
  • You need the maximum/minimum sum of a subarray
  • The condition is monotonic (if window [a,b] is valid, then [a,b+1] is either valid or invalid — not random)
  • You see keywords like "at most K", "at least K", "exactly K distinct"
  • Window validity can be checked by adding/removing one element at a time
Do NOT Use When
  • Subarray does NOT need to be contiguous (use DP/hashmap instead)
  • Problem asks for all possible subarrays (you need to enumerate, not optimize)
  • Window condition is not monotonic (adding an element can flip validity back and forth unpredictably)
  • You need to modify elements inside the window while processing
  • Problem is about ordering/permutation not about a contiguous range
Implementation Template — Line by Line

There are two main variants: Variable-size window (most common — window grows/shrinks based on validity) and Fixed-size window (window is always exactly K elements).

// === VARIABLE-SIZE SLIDING WINDOW (most common) ===
int left = 0;
int bestResult = 0;  // tracks best answer found so far

for (int right = 0; right < n; right++) {
    // STEP 1: EXPAND — add arr[right] to window
    // Update your state: sum += arr[right], or map.put(char, count), etc.
    windowState.add(arr[right]);

    // STEP 2: SHRINK — while window is INVALID, remove from left
    // This is a WHILE loop, not IF — keep shrinking until valid again
    while (windowIsInvalid(left, right)) {
        windowState.remove(arr[left]);  // undo what left contributed
        left++;                         // slide left boundary forward
    }

    // STEP 3: UPDATE — window is now valid, check if it's the best
    // Window size = right - left + 1
    bestResult = Math.max(bestResult, right - left + 1);
}
return bestResult;

// === FIXED-SIZE SLIDING WINDOW (window is always K elements) ===
for (int right = 0; right < n; right++) {
    windowState.add(arr[right]);              // add new element
    if (right >= K - 1) {                     // window reached size K
        // Process the window of exactly K elements
        bestResult = Math.max(bestResult, someCalculation());
        windowState.remove(arr[right - K + 1]); // remove leftmost for next step
    }
}
// === VARIABLE-SIZE SLIDING WINDOW ===
let left = 0, bestResult = 0;
for (let right = 0; right < n; right++) {
    windowState.add(arr[right]);        // EXPAND
    while (windowIsInvalid(left, right)) {
        windowState.remove(arr[left]);  // SHRINK
        left++;
    }
    bestResult = Math.max(bestResult, right - left + 1); // UPDATE
}
return bestResult;

// === FIXED-SIZE SLIDING WINDOW ===
for (let right = 0; right < n; right++) {
    windowState.add(arr[right]);
    if (right >= K - 1) {
        bestResult = Math.max(bestResult, someCalculation());
        windowState.remove(arr[right - K + 1]);
    }
}
Line-by-line understanding:
left = 0 → Window starts empty, left boundary at index 0
for right loop → right pointer keeps moving forward, expanding the window one element at a time
windowState.add(arr[right]) → incorporate the new element into our running state (sum, frequency map, etc.)
while (invalid) → KEY: this is a WHILE, not IF. The window might need multiple shrinks to become valid again
left++ → shrink from left side — we're discarding elements that make the window invalid
right - left + 1 → current window size. The +1 is because both indices are inclusive
Common Variations
Maximum Window
Keep expanding. Shrink only when invalid. Track max(window size) when valid.
Minimum Window
Shrink as much as possible while still valid. Track min(window size).
Fixed-Size Window
Window is always K elements. No shrink logic. Process at each position.
At Most K
Window valid when condition met ≤ K times. Standard shrink pattern.
Exact K
Trick: "exactly K" = "at most K" minus "at most K-1". Solve two sub-problems.
With Hashmap
Track character/item frequencies. Add/remove from map as window slides.

Solved Examples

Medium#3 — Longest Substring Without Repeating Characters
Approach

Map stores last index+1 of each character. When we see a duplicate inside the window, jump left to skip past it.

Step-by-step
1
Initialize left=0, maxLen=0, int[128] map for ASCII
2
Loop right: get char c. left = max(left, map[c]) to skip past duplicate
3
map[c] = right+1, maxLen = max(maxLen, right-left+1)
class Solution {
    public int lengthOfLongestSubstring(String s) {
        int[] map = new int[128];
        int left = 0, maxLen = 0;
        for (int right = 0; right < s.length(); right++) {
            char c = s.charAt(right);
            left = Math.max(left, map[c]);
            map[c] = right + 1;
            maxLen = Math.max(maxLen, right - left + 1);
        }
        return maxLen;
    }
}
var lengthOfLongestSubstring = function(s) {
    const map = new Map();
    let left = 0, maxLen = 0;
    for (let right = 0; right < s.length; right++) {
        const c = s[right];
        if (map.has(c)) left = Math.max(left, map.get(c));
        map.set(c, right + 1);
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
};
Medium#209 — Minimum Size Subarray Sum

Expand right, add to sum. While sum ≥ target: update minLen, subtract nums[left], left++.

class Solution {
    public int minSubArrayLen(int target, int[] nums) {
        int left = 0, sum = 0, minLen = Integer.MAX_VALUE;
        for (int right = 0; right < nums.length; right++) {
            sum += nums[right];
            while (sum >= target) {
                minLen = Math.min(minLen, right - left + 1);
                sum -= nums[left++];
            }
        }
        return minLen == Integer.MAX_VALUE ? 0 : minLen;
    }
}

    let left = 0, sum = 0, minLen = Infinity;

        sum += nums[right];
        while (sum >= target) { minLen = Math.min(minLen, right-left+1); sum -= nums[left++]; }
    }
    return minLen === Infinity ? 0 : minLen;

Two Pointers

Where Used

Sorted arrays, pairs/triplets with sum, palindrome checks, removing duplicates, in-place operations.

Hints to Identify

"sorted array""two sum""palindrome""pairs""in-place""O(1) space"
What is Two Pointers?

You place two fingers on the data structure (usually an array or string) and move them strategically to find an answer. Instead of using nested loops to compare every pair (O(n²)), you exploit sorted order or symmetry to eliminate half the search space each step (O(n)).

OPPOSITE-END (most common for sorted arrays):
Array: [1, 2, 3, 4, 5, 6, 7]
                        ↑                  ↑
                        left                right
  If sum too small → move left rightward (need bigger number)
  If sum too big   → move right leftward (need smaller number)

SAME-DIRECTION (for fast-slow, or in-place operations):
Array: [1, 2, 2, 3, 4, 4, 5]
         ↑        ↑
         read       write
  Both start at 0. Read pointer scans ahead. Write pointer only moves when we find something worth keeping.

The critical requirement: the array must be sorted (for opposite-end) OR the operation must have a directional property (for same-direction, like removing duplicates from sorted array).

When to Use • When NOT to Use
Use Two Pointers When
  • Array is sorted and you need pairs/triplets with a target sum
  • Checking palindromes (compare from both ends)
  • In-place modification needed (remove duplicates, move zeros)
  • You need to find a boundary in a partitioned array
  • Problem says "do it in O(1) extra space"
Do NOT Use When
  • Array is NOT sorted and you can't sort it (sorting changes indices)
  • You need all pairs, not just existence or optimal (still O(n²) worst case)
  • Elements have complex dependencies that prevent directional elimination
  • Working with linked lists with unknown length (use fast-slow instead)
Implementation Template
// === OPPOSITE-END TWO POINTERS (sorted array, target sum) ===
Arrays.sort(arr); // MUST sort first
int left = 0, right = arr.length - 1;
while (left < right) {
    int sum = arr[left] + arr[right];
    if (sum == target) {
        // Found a valid pair!
        // Process it, then move BOTH pointers
        left++; right--;
    } else if (sum < target) {
        left++;   // need a bigger sum → move left forward (bigger number)
    } else {
        right--;  // need a smaller sum → move right backward (smaller number)
    }
}

// === SAME-DIRECTION TWO POINTERS (in-place, e.g., remove duplicates) ===
int write = 0; // slow pointer: where to write next unique element
for (int read = 0; read < arr.length; read++) {
    if (arr[read] != arr[write]) {  // found a new unique element
        write++;
        arr[write] = arr[read];     // write it at the next position
    }
}
// All unique elements are now in arr[0..write]
// Return write + 1 as the new length
// OPPOSITE-END
arr.sort((a, b) => a - b);
let left = 0, right = arr.length - 1;
while (left < right) {
    const sum = arr[left] + arr[right];
    if (sum === target) { left++; right--; }
    else if (sum < target) left++;
    else right--;
}

// SAME-DIRECTION
let write = 0;
for (let read = 0; read < arr.length; read++) {
    if (arr[read] !== arr[write]) { write++; arr[write] = arr[read]; }
}
Why does opposite-end work on sorted arrays? If arr[left] + arr[right] is too small, moving right leftward would make it even smaller (array is sorted, so arr[right-1] ≤ arr[right]). The ONLY way to increase the sum is to move left forward. This directional property is what makes O(n) possible.
Common Variations
Two Sum (Sorted)
Classic opposite-end. Find pair summing to target in O(n).
Three Sum
Fix one element, then two-pointer on the rest. O(n²). Skip duplicates.
Palindrome Check
Compare s[left] vs s[right], move inward. O(n).
Remove Duplicates
Same-direction. Write pointer trails behind read pointer.
Container With Most Water
Opposite-end. Move shorter pointer (moving taller can't help).
Move Zeros
Same-direction. Write non-zero elements, fill rest with zeros.

Medium#11 — Container With Most Water

Two pointers at ends. Area = min(h[l], h[r]) × (r-l). Move the shorter pointer.

class Solution {
    public int maxArea(int[] height) {
        int left = 0, right = height.length - 1, maxWater = 0;
        while (left < right) {
            int h = Math.min(height[left], height[right]);
            maxWater = Math.max(maxWater, h * (right - left));
            if (height[left] < height[right]) left++; else right--;
        }
        return maxWater;
    }
}

Medium#15 — 3Sum

Sort. Fix i, two-pointer on rest for pairs summing to -nums[i]. Skip duplicates.

class Solution {
    public List<List<Integer>> threeSum(int[] nums) {
        Arrays.sort(nums);
        List<List<Integer>> res = new ArrayList<>();
        for (int i = 0; i < nums.length - 2; i++) {
            if (i > 0 && nums[i] == nums[i-1]) continue;
            int l = i + 1, r = nums.length - 1;
            while (l < r) {
                int sum = nums[i] + nums[l] + nums[r];
                if (sum == 0) {
                    res.add(Arrays.asList(nums[i], nums[l], nums[r]));
                    while (l < r && nums[l] == nums[l+1]) l++;
                    while (l < r && nums[r] == nums[r-1]) r--;
                    l++; r--;
                } else if (sum < 0) l++; else r--;
            }
        }
        return res;
    }
}
var threeSum = function(nums) {
    nums.sort((a, b) => a - b);
    const res = [];
    for (let i = 0; i < nums.length - 2; i++) {
        if (i > 0 && nums[i] === nums[i-1]) continue;
        let l = i + 1, r = nums.length - 1;
        while (l < r) {
            const sum = nums[i] + nums[l] + nums[r];
            if (sum === 0) {
                res.push([nums[i], nums[l], nums[r]]);
                while (l < r && nums[l] === nums[l+1]) l++;
                while (l < r && nums[r] === nums[r-1]) r--;
                l++; r--;
            } else if (sum < 0) l++; else r--;
        }
    }
    return res;
};

Fast & Slow Pointers

Where Used

Linked list cycle detection, finding middle element, detecting cycles in number sequences.

Hints to Identify

"cycle""loop""linked list""middle""happy number"
What is Fast & Slow Pointers?

Imagine two runners on a circular track. One runs at 2x speed (fast), the other at 1x speed (slow). If the track has a loop, the fast runner will eventually lap the slow runner — they'll meet at the same spot. If there's no loop (straight track), the fast runner just reaches the end first.

This is Floyd's Cycle Detection Algorithm. In code: slow moves 1 step per iteration, fast moves 2 steps. If they meet → cycle exists. If fast reaches null → no cycle. O(n) time, O(1) space (no hashmap needed).

Linked List: 1 → 2 → 3 → 4 → 5 → 3 (cycle back to 3)

Step 0: slow=1, fast=1
Step 1: slow=2, fast=3
Step 2: slow=3, fast=5
Step 3: slow=4, fast=4 ← THEY MEET! Cycle confirmed!

To find cycle START: reset slow to head, move both 1 step
Step 4: slow=1, fast=5
Step 5: slow=2, fast=3
Step 6: slow=3, fast=3 ← Cycle starts at node 3!

Why does finding the start work? When slow and fast meet inside the cycle, slow has traveled distance `a + b` (a = distance to cycle start, b = distance into cycle). Fast has traveled `a + b + k*c` (k = number of extra laps). Since fast = 2*slow: `a + b + kc = 2(a + b)` → `a = kc - b`. So if slow starts from head again and both move 1 step, they meet exactly at the cycle start after `a` steps.

When to Use • When NOT to Use
Use When
  • Detecting if a linked list has a cycle
  • Finding the start of a cycle
  • Finding the middle element of a linked list
  • Detecting cycles in number transformation sequences (happy number)
  • You need O(1) space — can't use a visited set
Do NOT Use When
  • Working with arrays (use index-based techniques instead)
  • You need to find the length of the cycle (fast-slow finds existence, use a counter from meeting point for length)
  • There are multiple possible cycles in a graph (fast-slow only works for single cycle in a linked list)
Implementation Template
ListNode slow = head, fast = head;
// PHASE 1: Detect if cycle exists
while (fast != null && fast.next != null) {
    slow = slow.next;         // moves 1 step
    fast = fast.next.next;    // moves 2 steps
    if (slow == fast) {       // they met → cycle!
        // PHASE 2: Find cycle start (optional)
        slow = head;           // reset slow to beginning
        while (slow != fast) {
            slow = slow.next;  // both move 1 step now
            fast = fast.next;
        }
        return slow;          // this IS the cycle start
    }
}
return null; // fast reached end → no cycle
let slow = head, fast = head;
while (fast && fast.next) {
    slow = slow.next;
    fast = fast.next.next;
    if (slow === fast) {
        slow = head;
        while (slow !== fast) { slow = slow.next; fast = fast.next; }
        return slow;
    }
}
return null;
Common Variations
Cycle Detection
Return true/false. Phase 1 only.
Cycle Start
Both phases. Reset slow to head after meeting.
Find Middle
When fast reaches end, slow is at middle. No cycle needed.
Number Sequence
Replace "next" with transformation function. Same logic.

Classic Floyd's. If fast reaches null, no cycle. If slow==fast, cycle.

            slow = slow.next; fast = fast.next.next;

Merge Intervals

Where Used

Overlapping intervals, meeting rooms, calendar scheduling.

Hints to Identify

"intervals""overlapping""merge""meeting rooms""schedule"
What is Merge Intervals?

You're given a list of ranges (intervals) on a number line. Many of them overlap. Your job is to combine overlapping intervals into single, larger intervals. The trick: sort by start time first, then any overlaps become adjacent and easy to detect.

Before sort: [1,3], [8,10], [2,6], [15,18]
After sort:  [1,3], [2,6], [8,10], [15,18]

Process [1,3]: result = [[1,3]]
Process [2,6]: 2 <= 3? YES overlap! → merge → result = [[1,6]]
Process [8,10]: 8 <= 6? NO → new interval → result = [[1,6], [8,10]]
Process [15,18]: 15 <= 10? NO → new → result = [[1,6], [8,10], [15,18]]

Overlapping condition: current.start <= last_merged.end
When to Use • When NOT to Use
Use When
  • Problem gives you intervals/ranges with start and end values
  • You need to find overlapping regions
  • Scheduling problems: meeting rooms, calendars
  • You need to insert an interval and merge the result
Do NOT Use When
  • Intervals are not on a 1D number line (2D ranges need different approach)
  • You need to find intersections of two separate interval lists (different template)
  • Problem is about selecting non-overlapping subsets with max total value (that's weighted interval scheduling — use DP)
Implementation Template
Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0])); // SORT BY START
List<int[]> result = new ArrayList<>();
for (int[] iv : intervals) {
    if (result.isEmpty() || result.get(result.size()-1)[1] < iv[0]) {
        result.add(iv);  // NO overlap → add as new interval
    } else {
        // OVERLAP → merge by extending the end
        result.get(result.size()-1)[1] = Math.max(
            result.get(result.size()-1)[1], iv[1]);
    }
}
intervals.sort((a, b) => a[0] - b[0]);
const result = [];
for (const iv of intervals) {
    if (!result.length || result[result.length-1][1] < iv[0]) result.push(iv);
    else result[result.length-1][1] = Math.max(result[result.length-1][1], iv[1]);
}
Key detail: the overlap check is `last.end < current.start` for NO overlap. If `current.start <= last.end`, they overlap. We take `max(last.end, current.end)` because the current interval might extend beyond the last merged one. We take max, not current.end directly.
Common Variations
Merge All Overlaps
Standard. Sort by start, merge adjacent overlapping.
Remove Min to Make Non-Overlapping
Sort by END. Greedy pick earliest-ending. Count removals.
Insert Interval
Find insertion point, add, then merge. Or: collect before/after separately.
Meeting Rooms (min rooms)
Sort start times and end times separately. Two-pointer to count concurrent meetings.

Cyclic Sort

Where Used

Arrays with numbers in range [1,n]. Missing numbers, duplicates, first missing positive.

Hints to Identify

"range 1 to n""missing number""duplicate""O(1) extra space"
What is Cyclic Sort?

When you know the array contains numbers from 1 to n (or 0 to n-1), you know exactly where each number should be: the number 3 should be at index 2, the number 7 should be at index 6, etc. Cyclic sort exploits this: for each position, swap the element to its correct position, and keep doing it until the current position has the right number.

nums = [3, 1, 5, 4, 2]  (range 1-5, n=5)

i=0: nums[0]=3, should be at index 2. Swap(0,2) → [5, 1, 3, 4, 2]
i=0: nums[0]=5, should be at index 4. Swap(0,4) → [2, 1, 3, 4, 5]
i=0: nums[0]=2, should be at index 1. Swap(0,1) → [1, 2, 3, 4, 5]
i=0: nums[0]=1 ✓ correct! Move on.
i=1: nums[1]=2 ✓ correct!
i=2: nums[2]=3 ✓ correct!
...all correct. Sorted in O(n) without extra space!
When to Use • When NOT to Use
Use When
  • Numbers are in a known range [1,n] or [0,n-1]
  • You need missing numbers, duplicates, or misplaced elements
  • Problem requires O(1) extra space
Do NOT Use When
  • Numbers have arbitrary values outside a predictable range
  • There are negative numbers (unless you can offset them)
  • Array contains duplicates that prevent unique placement (handle carefully — use negation marking instead)
Implementation Template
int i = 0;
while (i < nums.length) {

What is In-place Linked List Reversal?

In a linked list, each node points to the next. To reverse, you need to flip every arrow. The challenge: once you change `node.next`, you've lost the rest of the list. So you MUST save the next node before breaking the link.

Before: 1 → 2 → 3 → 4 → null

prev=null, curr=1:
  next = 2 (save!)    1.next = null (reverse!)   prev=1, curr=2
  null ← 1   2 → 3 → 4 → null

prev=1, curr=2:
  next = 3 (save!)    2.next = 1 (reverse!)   prev=2, curr=3
  null ← 1 ← 2   3 → 4 → null

... continue ...
Result: null ← 1 ← 2 ← 3 ← 4, return prev=4

The three-pointer dance: prev (what the current node should point to), curr (node we're currently reversing), next (saved so we don't lose the rest of the list).

When to Use • When NOT to Use
Use When
  • Problem asks to reverse a linked list (fully or partially)
  • Palindrome check for linked list (reverse half, compare)
  • K-group reversal
  • Swap adjacent pairs in a linked list
Do NOT Use When
  • Working with arrays (just swap elements, no pointer manipulation)
  • You can create new nodes (simpler but uses O(n) space — sometimes acceptable)
Implementation Template
ListNode prev = null, curr = head;
while (curr != null) {
    ListNode next = curr.next;  // 1. SAVE next before breaking link
    curr.next = prev;           // 2. REVERSE: point current to previous
    prev = curr;                // 3. ADVANCE prev
    curr = next;                // 4. ADVANCE curr using saved next
}
return prev; // prev is now the new head
let prev = null, curr = head;

    const next = curr.next;  // SAVE
    curr.next = prev;        // REVERSE
    prev = curr;             // ADVANCE prev
    curr = next;             // ADVANCE curr
}
return prev;
The order matters: If you do `curr.next = prev` BEFORE saving next, you've lost the reference to the rest of the list. Always save next FIRST.

What is Tree BFS?

BFS (Breadth-First Search) explores a tree level by level, left to right. Think of it as pouring water on the root — it spreads to all children first (level 1), then to all grandchildren (level 2), etc. You use a Queue (FIFO) to implement this: enqueue root, then repeatedly dequeue a node, process it, and enqueue its children.

Tree:    3
        / \
      9   20
    /    / \
  15   7

Queue: [3]              → dequeue 3, enqueue 9, 20
Queue: [9, 20]           → dequeue 9, enqueue 15, 7
Queue: [20, 15, 7]        → dequeue 20 (no children)
Queue: [15, 7]            → dequeue 15 (no children)
Queue: [7]                  → dequeue 7 (no children)
Queue: []                   → done

Levels: [[3], [9, 20], [15, 7]]

Why use `size` variable? Inside the while loop, you capture `queue.size()` BEFORE processing. This tells you exactly how many nodes are at the current level. You process exactly that many nodes. This is what lets you group results by level.

When to Use • When NOT to Use
Use When
  • Problem asks for level-by-level processing
  • You need shortest path in an unweighted tree/graph
  • You need to process nodes in order of distance from root
  • Zigzag traversal, right-side view, etc.
Do NOT Use When
  • You need path information (use DFS with path parameter)
  • You need to compare subtrees (use DFS with return values)
  • The tree is very deep but narrow (BFS queue gets large; DFS uses O(depth) stack)
  • You need bottom-up computation like max path sum (use post-order DFS)
Implementation Template

        if (node.right != null) q.offer(node.right);

Tree DFS

Where Used

Pre/In/Post-order traversal, path sum, max path sum, diameter, validating BST, all bottom-up or top-down processing.

Hints to Identify

"path sum""diameter""height""pre/in/post order""BST property""max path"
What is Tree DFS?

DFS (Depth-First Search) goes as deep as possible along one branch before backtracking. Think of it like exploring a maze: you go down one path until you hit a dead end, then back up and try the next path. In a tree, you use recursion (the call stack acts as your path tracker).

There are three traversal orders, defined by WHEN you process the current node relative to its children:

Tree:    1
       / \
     2   3
  / \
 4  5

PRE-ORDER (process node FIRST, then children):
  Visit: 1 → 2 → 4 → 5 → 3
  Use when: copying a tree, serializing, prefix expressions

IN-ORDER (process LEFT, then node, then RIGHT):
  Visit: 4 → 2 → 5 → 1 → 3
  Use when: BST operations (gives sorted order!), expression trees

POST-ORDER (process children FIRST, then node):
  Visit: 4 → 5 → 2 → 3 → 1
  Use when: deleting a tree, calculating size/height (need children's results first), bottom-up DP on trees

The two modes of DFS:
• Top-down (pre-order with parameters): Pass info DOWN the tree. Example: pass the current path sum as a parameter.
• Bottom-up (post-order with return values): Collect info FROM children, combine at current node, return upward. Example: return the height of a subtree so the parent can compute diameter.

When to Use • When NOT to Use
Use When
  • You need height, diameter, path sum — these require bottom-up info from children
  • Problem asks to validate BST (need to pass min/max constraints down)
  • You need all root-to-leaf paths (top-down with path accumulator)
  • Post-order computation: max path sum, subtree properties
  • Problem involves subtree comparison (are two subtrees identical/mirror?)
Do NOT Use When
  • You need level-by-level processing (use BFS instead)
  • You need shortest path in unweighted tree (BFS guarantees shortest)
  • The tree is extremely deep and you risk stack overflow (use iterative DFS with explicit stack)

Bottom-up. For each node, compute the maximum "gain" you can get by starting a path from this node going downward (you can only take one branch — left or right, not both, because paths can't branch).     public int maxPathSum(TreeNode root) {
        gain(root);
        return maxSum;
    }
    private int gain(TreeNode node) {
        if (node == null) return 0;

    }

What is Backtracking?

Backtracking is a systematic way to explore ALL possible arrangements by making a choice, recursing deeper, and then undoing that choice (backtracking) to try the next option. It's like solving a maze by trying every path: go forward, if you hit a dead end, go back to the last fork and try the other direction.

Generating permutations of [1, 2, 3]:

bt([]):
  Choose 1: bt([1])
    Choose 2: bt([1,2])
      Choose 3: bt([1,2,3]) → SAVE [1,2,3] → backtrack
      No more choices. backtrack
    Choose 3: bt([1,3])
      Choose 2: bt([1,3,2]) → SAVE [1,3,2] → backtrack
    No more choices. backtrack
  Choose 2: bt([2])
    Choose 1: bt([2,1]) → ... → SAVE [2,1,3]
    Choose 3: bt([2,3]) → ... → SAVE [2,3,1]
  Choose 3: bt([3])
    Choose 1: bt([3,1]) → ... → SAVE [3,1,2]
    Choose 2: bt([3,2]) → ... → SAVE [3,2,1]

Total: 6 permutations (3! = 6)

The three steps that repeat at every level of recursion:
1. CHOOSE — pick an option and add it to your current state (path)
2. EXPLORE — recurse to go deeper with this choice
3. UNDO — remove the choice (backtrack) so the next iteration can try a different option

When to Use • When NOT to Use
Use When
  • Problem asks for ALL possible arrangements/combinations/subsets
  • You need to try every possibility and collect valid ones
  • There's a constraint that invalidates certain choices (pruning)
  • Problem has a combinatorial structure: N-Queens, Sudoku, word search
Do NOT Use When
  • You only need ONE solution (use greedy if it works, or DP for optimal)
  • Input size is very large (backtracking is exponential — 2^n, n!)
  • There's a polynomial-time solution (DP, greedy) — backtracking would TLE
  • Problem asks for the COUNT not the actual arrangements (often DP is better)
Implementation Template

Critical details:
• Save a COPY of the path, not the reference. In Java: `new ArrayList<>(path)`. In JS: `[...path]`. Otherwise the list gets modified later and all entries point to the same object.
• `start` vs `i`: For combinations (no reuse), pass `i+1`. For permutations (reuse), pass `0`. For combinations with unlimited reuse (coin change), pass `i` (not i+1).
• Pruning: The `isValidChoice` check is what makes backtracking fast enough. Without pruning, you explore the entire search space. With pruning, you cut branches early.

What is the Top K Pattern?

When you need the K largest/smallest/most frequent elements from a dataset, you DON'T need to sort the entire array (O(n log n)). Instead, maintain a heap of size K. The trick is choosing the RIGHT type of heap:

RULE: Use the OPPOSITE heap type

Top K LARGEST → use MIN-HEAP of size K
  Why? The smallest of the top K gets evicted when a bigger element comes in.
  Heap: [5, 8, 10] (min=5). New element 12 comes → evict 5 → [8, 10, 12]

Top K SMALLEST → use MAX-HEAP of size K
  Why? The largest of the top K gets evicted when a smaller element comes in.
  Heap: [1, 3, 5] (max=5). New element 2 comes → evict 5 → [1, 2, 3]

Kth LARGEST → same as top K largest → MIN-HEAP of size K
  After processing all elements, heap.peek() = Kth largest

Time: O(n log K) instead of O(n log n). Space: O(K)
When to Use • When NOT to Use
Use When
  • You need top K or Kth element by some criterion
  • K is much smaller than n (then O(n log K) << O(n log n))
  • Elements have a natural ordering (frequency, value, distance)
Do NOT Use When
  • K is close to n (then O(n log K) ≈ O(n log n), just sort)
  • You need all elements sorted, not just top K
  • The ordering criterion is complex and multi-dimensional (might need custom data structure)
Implementation Template

What is Graph Traversal?

A graph is a set of nodes connected by edges. Unlike trees, graphs can have cycles, and nodes can be connected in any pattern. The two fundamental ways to explore a graph are BFS and DFS — same concepts as in trees, but with a visited set to prevent infinite loops (since graphs can have cycles).

Step 1: Build adjacency list from edges
edges = [[0,1],[1,2],[0,2],[3,4]]
adj[0] = [1, 2], adj[1] = [0, 2], adj[2] = [0, 1], adj[3] = [4], adj[4] = [3]

Step 2: Choose BFS or DFS
BFS (Queue): explores level by level → shortest path in unweighted graph
DFS (Recursion/Stack): goes deep first → connectivity, cycle detection, flood fill

Step 3: Always track visited[]
Without visited[], BFS/DFS loops forever on cyclic graphs.
visited[0]=true, visited[1]=true, ... → only process unvisited nodes.

Grid as graph: A 2D grid is a graph where each cell is a node connected to its 4 (or 8) neighbors. "Island" problems are just "connected components" on a grid graph. Use DFS to flood-fill, or BFS for shortest distance.

When to Use • When NOT to Use
Use When
  • Problem mentions nodes, edges, neighbors
  • Shortest path in unweighted graph (BFS)
  • Connected components (count islands, friend circles)
  • Topological ordering (course schedule with prerequisites)
  • 2D grid problems with "island", "flood fill", "rotting"
Do NOT Use When
  • Graph is a tree (use tree-specific patterns — simpler)
  • You need shortest path with weighted edges (use Dijkstra — different algorithm)
  • Problem is really about arrays/strings disguised (check carefully)

What is the Sorting Pattern?

Many problems become trivial once the data is in the right order. The sorting pattern is about defining what "comes before what" using a custom comparator. After sorting, the answer often becomes a linear scan or two-pointer problem.

Custom Comparator Logic:
compare(a, b):
  return negative → a comes before b
  return positive → b comes before a
  return 0 → equal

Example: #179 Largest Number
Compare "3" and "30": should "330" or "303" come first?
"330" > "303" → "3" should come before "30"
→ compare("3","30") = "303".compareTo("330") < 0 → correct!

Example: #75 Sort Colors (Dutch National Flag)
Not a comparator problem — it's a single-pass partition.
Three regions: [0...low]=0s, [low...mid]=1s, [mid...high]=unknown, [high...end]=2s

  • Values have a bounded range (use counting sort for O(n))

Do NOT Use When

  • There's a linear-time solution without sorting (don't sort unnecessarily)

What is Binary Search?

Like looking up a word in a dictionary: you open to the middle, decide if the word is before or after, and discard half the book. Repeat until found. This turns O(n) linear search into O(log n).

Array: [1, 3, 5, 7, 9, 11, 13], target = 7

left=0, right=6, mid=3, nums[3]=7 → FOUND!

If target=8:
left=0, right=6, mid=3, nums[3]=7 < 8 → left=4
left=4, right=6, mid=5, nums[5]=11 > 8 → right=4
left=4, right=4, mid=4, nums[4]=9 > 8 → right=3
left(4) > right(3) → NOT FOUND (return -1)

Three templates exist for different use cases:
1. Exact match: while(left <= right) → standard
2. Left boundary: while(left < right) → no +1 in mid for right
3. Right boundary: while(left < right) → +1 in mid for left

Binary search can also be used on answer spaces. Not just arrays! If the answer is monotonic (e.g., "minimum speed to finish in H hours"), you can binary search on the SPEED value, not an array index. The "array" is the range [minSpeed, maxSpeed]. Check if a speed works → adjust bounds.

When to Use • When NOT to Use
Use When
  • Array is sorted (or partially sorted like rotated arrays)
  • You need O(log n) time
  • The condition has a single transition point (monotonic: true...true...false...false)
  • Search space reduction problems (minimize X such that condition holds)
Do NOT Use When
  • Array is not sorted and not monotonic in any way
  • You need the first/last occurrence but haven't mastered the boundary templates (use linear to be safe)
  • Problem is simpler with linear scan and n is small
Implementation Template

Fix: the right-boundary template uses (left+right+1)/2 so mid is biased rightward. The left-boundary template avoids the issue because right=mid shrinks correctly.

 3   5     ← 1 is on top (smallest)
insert(2) → 2 bubbles up to correct position
poll() → removes 1, replaces with last element, bubbles down

MAX-HEAP (Java: reverse comparator):
    5
  /  \
 3   1     ← 5 is on top (largest)

Key operations: O(log n) for insert/poll, O(1) for peek
Build heap from array: O(n) (not n log n!)

When is this different from Top K pattern? Top K is a specific use case (fixed-size heap). Priority Queue pattern is broader: merge K lists, streaming median, task scheduling — where the heap size varies and the logic is more complex than just "keep K elements."

When to Use • When NOT to Use
Use When
  • You need to always access the min/max element dynamically
  • Merge K sorted streams (always pick the smallest next element)
  • Streaming median (maintain two heaps: max for lower half, min for upper half)
  • Task scheduling (always pick highest priority task)
  • Dijkstra's shortest path (always expand the closest unvisited node)
Do NOT Use When
  • You just need top K elements (use the simpler Top K pattern)
  • The ordering is static (sort once, no need for dynamic access)
  • You're working with JavaScript (no built-in heap — must simulate with array or use library)

Push all heads into min-heap. Repeatedly pop min, append to result, push next.

class Solution {
    public ListNode mergeKLists(ListNode[] lists) {
        PriorityQueue<ListNode> pq = new PriorityQueue<>((a, b) -> a.val - b.val);
        for (ListNode n : lists) if (n != null) pq.offer(n);
        ListNode dummy = new ListNode(0), tail = dummy;
        while (!pq.isEmpty()) {
            ListNode min = pq.poll();
            tail.next = min; tail = tail.next;
            if (min.next != null) pq.offer(min.next);
        }
        return dummy.next;
    }
}

Greedy Approach

Where Used

Interval scheduling, jump games, stock trading, minimum platforms, maximizing with local optimal choices.

Hints to Identify

What is the Greedy Approach?

At every decision point, make the locally optimal choice (what seems best right now) and never look back. No recursion, no DP table, no consideration of future consequences. If the local optimum happens to lead to the global optimum, greedy works. The hard part is proving it does.

Activity Selection (the classic greedy proof):
Activities: [A:0-6], [B:1-4], [C:3-5], [D:5-7], [E:8-9]
Sorted by end time: B(4), C(5), A(6), D(7), E(9)

Pick B(0-4) → can we pick C(3-5)? NO, overlaps → skip
Pick A(0-6)? NO, overlaps → skip
Pick D(5-7)? YES → no overlap with B. Pick it.
Pick E(8-9)? YES → no overlap with D. Pick it.
Selected: B, D, E → 3 activities (maximum possible!)

Why it works: Picking the activity that ends earliest
leaves the MOST time for remaining activities. This is provably optimal.

Jump Game intuition:
At position i, you can jump to any position in [i+1, i+nums[i]].
Greedy: track the FARTHEST position reachable. If you can reach position i,
you can also reach everything between previous maxReach and i+nums[i].
So just track one number: maxReach = max(maxReach, i + nums[i]).

How to know if greedy works? Ask: "If I make the best choice now, could a worse choice lead to a better overall result?" If you can construct a counterexample → greedy doesn't work, use DP. If you can't → greedy likely works. In interviews, when in doubt, try greedy first (simpler code), then verify with examples.

When to Use • When NOT to Use
Use When
  • Problem has a "greedy choice property" — local optimum = global optimum
  • Interval scheduling (sort by end, pick non-overlapping)
  • Jump game (always jump as far as possible)
  • Stock trading (single transaction) (buy low, sell high — one pass)
  • Problem can be solved by sorting first then making sequential choices
  • You need maximum count/feasibility (not maximum value of subset)
Do NOT Use When
  • Coin change (general case) — [1,3,4] target=6: greedy picks 4+1+1=3 coins, but optimal is 3+3=2 coins
  • 0/1 Knapsack — taking an item excludes others; need DP for combinations
  • You can find a counterexample where a non-greedy choice is better
  • Problem involves subsets with optimal total value (not just count)

class Solution {
    public int jump(int[] nums) {
        int jumps = 0, currentEnd = 0, farthest = 0;
        for (int i = 0; i < nums.length - 1; i++) {
            farthest = Math.max(farthest, i + nums[i]);
            if (i == currentEnd) {
                jumps++;
                currentEnd = farthest;
            }
        }
        return jumps;
    }
}

Divide & Conquer

Where Used

Merge sort, quick sort, counting inversions, maximum subarray, power computation, closest pair of points.

Hints to Identify

"divide""conquer""merge""inversions""O(n log n)""split in half"
What is Divide & Conquer?

Break the problem into smaller subproblems of the same type, solve them recursively, then combine the results. It's the algorithmic equivalent of "divide and rule" — break a big problem into smaller ones you can solve easily.

The Three Steps:

1. DIVIDE: Split the problem in half
  Array[0..n-1] → Array[0..mid-1] + Array[mid..n-1]

2. CONQUER: Solve each half recursively
  leftResult = solve(leftHalf)
  rightResult = solve(rightHalf)
  (Base case: when problem is small enough, solve directly)

3. COMBINE: Merge the two results
  result = combine(leftResult, rightResult)

Time Complexity (Master Theorem):
T(n) = 2·T(n/2) + O(n) → O(n log n)
(The O(n) is the combine step — usually a linear merge)

What makes D&C different from plain recursion? In plain recursion, you call yourself on a slightly smaller input (n-1). In D&C, you split the input roughly in HALF (n/2), which gives the log n factor. This log n is what separates O(n log n) algorithms from O(n²).

When to Use • When NOT to Use
Use When
  • The problem can be naturally split in half (arrays, strings, matrices)
  • You need O(n log n) time (faster than O(n²))
  • Subproblems are independent (solving left half doesn't affect right half)
  • The combine step is straightforward (merge, compare, add)
  • Classic examples: merge sort, quicksort, counting inversions, max subarray, power(x,n)
Do NOT Use When
  • Subproblems are NOT independent (e.g., Fibonacci — fib(n) depends on fib(n-1) AND fib(n-2), but they overlap — use memoization/DP instead)
  • There's a simpler O(n) or O(n²) solution that's fast enough
  • The problem can't be cleanly split into independent subproblems
Implementation Template
// GENERIC DIVIDE & CONQUER TEMPLATE
Result solve(Problem problem) {
    // BASE CASE
    if (problem.isSmallEnough()) return bruteForce(problem);

    // DIVIDE: split into subproblems
    SubProblem leftHalf = problem.leftHalf();
    SubProblem rightHalf = problem.rightHalf();

    // CONQUER: solve subproblems recursively
    Result leftResult = solve(leftHalf);
    Result rightResult = solve(rightHalf);

    return combine(leftResult, rightResult);

// EXAMPLE: Max Subarray using D&C
int maxSubarray(int[] nums, int left, int right) {
    if (left == right) return nums[left]; // BASE

    int mid = left + (right - left) / 2;
    int leftMax = maxSubarray(nums, left, mid);     // CONQUER left
    int rightMax = maxSubarray(nums, mid + 1, right);  // CONQUER right

    int crossMax = crossMid(nums, left, mid, right);    // CROSSING subarray
    return Math.max(Math.max(leftMax, rightMax), crossMax);
}

int crossMid(int[] nums, int left, int mid, int right) {
    int leftSum = Integer.MIN_VALUE, sum = 0;
    for (int i = mid; i >= left; i--) { sum += nums[i]; leftSum = Math.max(leftSum, sum); }
    int rightSum = Integer.MIN_VALUE; sum = 0;
    for (int i = mid + 1; i <= right; i++) { sum += nums[i]; rightSum = Math.max(rightSum, sum); }
    return leftSum + rightSum;
}

Approach

Split at mid. Max subarray is entirely in left, entirely in right, or crosses the midpoint. Solve left and right recursively. For crossing: compute max sum extending left from mid and right from mid+1, combine.

class Solution {
    public int maxSubArray(int[] nums) {
        return divide(nums, 0, nums.length - 1);
    }
    private int divide(int[] nums, int left, int right) {
        if (left == right) return nums[left];
        int mid = left + (right - left) / 2;
        int leftMax = divide(nums, left, mid);
        int rightMax = divide(nums, mid + 1, right);
        int crossMax = cross(nums, left, mid, right);
        return Math.max(Math.max(leftMax, rightMax), crossMax);
    }
    private int cross(int[] nums, int left, int mid, int right) {
        int leftSum = Integer.MIN_VALUE, sum = 0;
        for (int i = mid; i >= left; i--) { sum += nums[i]; leftSum = Math.max(leftSum, sum); }
        int rightSum = Integer.MIN_VALUE; sum = 0;
        for (int i = mid + 1; i <= right; i++) { sum += nums[i]; rightSum = Math.max(rightSum, sum); }
        return leftSum + rightSum;
    }
}

Hard#493 — Reverse Pairs
Approach

Modified merge sort. During the merge step (before actually merging), count "split inversions": pairs (i, j) where i is in the left half, j is in the right half, and nums[i] > 2*nums[j]. These are inversions that span the split point. The within-half inversions are counted recursively.

class Solution {
    public int reversePairs(int[] nums) {
        return mergeSort(nums, 0, nums.length - 1);
    }
    private int mergeSort(int[] nums, int left, int right) {
        if (left >= right) return 0;
        int mid = left + (right - left) / 2;
        int count = mergeSort(nums, left, mid) + mergeSort(nums, mid + 1, right);
        // Count split inversions before merging
        int j = mid + 1;
        for (int i = left; i <= mid; i++) {
            while (j <= right && (long) nums[i] > 2L * nums[j]) j++;
            count += (j - (mid + 1));
        }
        merge(nums, left, mid, right);
        return count;
    }
    private void merge(int[] nums, int left, int mid, int right) {
        int[] temp = new int[right - left + 1];
        int i = left, j = mid + 1, k = 0;
        while (i <= mid && j <= right)
            temp[k++] = nums[i] <= nums[j] ? nums[i++] : nums[j++];
        while (i <= mid) temp[k++] = nums[i++];
        while (j <= right) temp[k++] = nums[j++];
        System.arraycopy(temp, 0, nums, left, temp.length);
    }
}
var reversePairs = function(nums) {
    const mergeSort = (l, r) => {
        if (l >= r) return 0;
        const mid = Math.floor((l + r) / 2);
        let count = mergeSort(l, mid) + mergeSort(mid + 1, r);
        let j = mid + 1;
        for (let i = l; i <= mid; i++) {
            while (j <= r && nums[i] > 2 * nums[j]) j++;
            count += j - (mid + 1);
        }
        const temp = [];
        let i2 = l, j2 = mid + 1;
        while (i2 <= mid && j2 <= r) temp.push(nums[i2++] || nums[j2++]);
        while (i2 <= mid) temp.push(nums[i2++]);
        while (j2 <= r) temp.push(nums[j2++]);
        for (let k = 0; k < temp.length; k++) nums[l + k] = temp[k];
        return count;
    };
    return mergeSort(0, nums.length - 1);
};
